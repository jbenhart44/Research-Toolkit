# PCV Research Protocol — Parallel Planning with Instrumentation

> **When to use**: You have a complex project charge and want to study how different planning strategies (depth-first vs breadth-first) produce different designs. Research-grade protocol — spawns 4+ subagents, significant token cost.
>
> **Cost warning**: This command spawns ~14 subagent calls across 4 parallel instances. Estimated cost: 150K-300K tokens per run depending on charge complexity. If you are on a usage-based API plan, expect this to be one of the most expensive commands in the toolkit.

You are executing the PCV Research Protocol. This is a **research instrument** that runs the PCV planning phase (charge → clarification → MakePlan) through two different agent-answering strategies, each replicated twice, to study how AI agents navigate planning decisions and where they converge or diverge.

**Version**: 1.0 (2026-03-31)

**Purpose**: Generate comparative data on depth-first vs. breadth-first agent-driven PCV planning for:
1. Evidence on structured verification effectiveness
2. Research methodology contribution (how do AI agents plan?)
3. Token/quality tradeoff analysis

## ARCHITECTURE OVERVIEW

```
                        CHARGE FILE
                            |
              +-------------+-------------+
              |                           |
        DEPTH-FIRST (x2)          BREADTH-FIRST (x2)
              |                           |
      +-------+-------+          +-------+-------+
      |               |          |               |
   Instance 1    Instance 2   Instance 3    Instance 4
      |               |          |               |
  [A answers]    [A answers]  [A answers]    [A answers]
  [B answers]    [B answers]  [B answers]    [B answers]
  [C decides]    [C decides]  [C decides     [C decides
   (at end)]     (at end)]    per question]   per question]
      |               |          |               |
   MakePlan-D1   MakePlan-D2  MakePlan-B1   MakePlan-B2
      |               |          |               |
      +-------+-------+          +-------+-------+
              |                           |
     DF Convergence Report      BF Convergence Report
              |                           |
              +-------------+-------------+
                            |
                  CROSS-MODE COMPARISON
                  (DF vs BF analysis)
                            |
                    REPORT TO HUMAN
                  + instrumentation log
```

### Agent Roles (per instance)

| Agent | Role | Context |
|-------|------|---------|
| **Agent A (Proposer)** | Answers each clarification question independently | Sees charge + question only. No access to Agent B's answers. |
| **Agent B (Proposer)** | Answers each clarification question independently | Sees charge + question only. No access to Agent A's answers. |
| **Agent C (Decider)** | Reviews both proposals and makes the final answer | Sees charge + question + both proposals + its own judgment. |

### Two Modes

**Depth-First**: Agents A and B each answer ALL clarification questions independently (no cross-visibility). Agent C reviews all paired answers at the end and produces a single resolved answer set. The MakePlan is drafted from Agent C's resolved answers.

**Breadth-First**: For each question in sequence, Agents A and B answer independently, then Agent C decides. Agents A and B see the resolved answer (Agent C's decision) before the next question is posed. Context accumulates — later answers are informed by earlier resolutions.

### 0g: Mode Recommendation (Optional)

Before launching instances, assess the charge type and recommend a configuration:

| Charge Type | Recommended Config | Rationale |
|-------------|-------------------|-----------|
| Well-scoped engineering (tight constraints, independent questions) | DF-only (2 instances) | Independent questions don't benefit from context accumulation |
| Cascading architecture (interdependent design decisions) | BF-only (2 instances) | Later answers depend on earlier resolutions |
| Analysis/report (moderate interdependency) | Full 2+2 | Default — both modes add value |
| Presentation/slides (many independent design decisions) | DF-only (2 instances) | Design decisions are mostly independent |
| Under-specified charge (convergence likely <50%) | **Rewrite charge first** | Low convergence signals an unclear charge |
| Highly specified charge (convergence likely >85%) | Single instance sufficient | Mode doesn't matter when the answer space is narrow |

## STEP 1: RUN DEPTH-FIRST INSTANCES (parallel, launched simultaneously with STEP 2)

Spawn **two independent depth-first instances in parallel** using the Agent tool. These launch at the same time as the breadth-first instances in STEP 2 — all four instances run concurrently.

### Instructions for Each Depth-First Instance

Each instance receives the charge file path and the master question list. For each instance, the agent must:

1. **Read the charge file** from disk.
2. **Read the master question list** from disk.
3. **For each question**, spawn Agent A and Agent B **in parallel**:

   **Agent A prompt:**
   > You are Agent A (Proposer) in a PCV Research experiment. You are answering a planning clarification question for the project described in the charge file.
   >
   > **Charge file contents:** [insert charge content]
   >
   > **Question [N] of [total]:** [question text]
   >
   > **Context:** [brief context for why this question matters, from the question list]
   >
   > Answer this question as if you were the human project owner making a strategic decision. Be specific and decisive — do not hedge. Explain your reasoning in 2-4 sentences, then state your answer clearly.
   >
   > You are working independently. You will NOT see Agent B's answer.

   **Agent B prompt:**
   > [Same as Agent A, but:]
   > You are Agent B (Proposer). You are working independently. You will NOT see Agent A's answer.
   >
   > **APPROACH DIFFERENTLY:** Before answering, consider at least one alternative interpretation or answer. If the alternative is clearly inferior, use the standard answer — but document why you rejected the alternative. Your value comes from offering a different perspective.

4. **After ALL questions are answered by both A and B**, spawn Agent C:

   **Agent C prompt (depth-first):**
   > You are Agent C (Decider) in a PCV Research experiment. Two independent agents answered [N] planning clarification questions for a project. Your job is to review their paired answers and produce a single resolved answer for each question.
   >
   > **Charge file contents:** [insert charge content]
   >
   > **For each question, you receive:**
   > - The question text
   > - Agent A's answer
   > - Agent B's answer
   >
   > [Insert all Q&A pairs]
   >
   > **For each question, produce:**
   > 1. Your resolved answer (may agree with A, B, a synthesis, or your own different answer)
   > 2. Brief reasoning (which agent was stronger and why, or why you chose differently)
   > 3. Convergence tag: AGREE (A and B said essentially the same thing), DIVERGE (meaningfully different answers), or PARTIAL (some overlap, some difference)

5. **Draft the MakePlan** using Agent C's resolved answers, following PCV v3.9 MakePlan structure (P5).

6. **Save all artifacts** to the instance directory.

7. **Return**: the MakePlan content, Agent C's convergence tags, and a summary of decisions made.

**Log each agent invocation to instrumentation.csv** with timestamp, estimated tokens, and wall clock time.

## STEP 2: RUN BREADTH-FIRST INSTANCES (parallel, launched simultaneously with STEP 1)

Spawn **two independent breadth-first instances in parallel** using the Agent tool. These launch at the same time as the depth-first instances in STEP 1 — all four instances run concurrently. STEP 3 begins only after all four instances (both DF and BF) have completed.

### Instructions for Each Breadth-First Instance

Each instance receives the charge file path and the master question list. The key difference from depth-first: **questions are answered one at a time, and all agents see previous resolutions.**

For each instance, the agent must:

1. **Read the charge file** from disk.
2. **Read the master question list** from disk.
3. **Initialize a shared context accumulator**: starts empty, grows with each resolved answer.
4. **For each question IN ORDER:**

   a. Spawn Agent A and Agent B **in parallel**:

   **Agent A prompt:**
   > You are Agent A (Proposer) in a PCV Research experiment (breadth-first mode).
   >
   > **Charge file contents:** [insert charge content]
   >
   > **Previously resolved answers:**
   > [Insert all prior Q&A resolutions from Agent C, or "None yet" if Q1]
   >
   > **Current Question [N] of [total]:** [question text]
   >
   > **Context:** [why this question matters]
   >
   > Answer this question considering the previously resolved answers. Be specific and decisive. Explain reasoning in 2-4 sentences, then state your answer clearly.

   **Agent B prompt:**
   > [Same structure as Agent A, plus the APPROACH DIFFERENTLY instruction from depth-first]

   b. **Spawn Agent C** with both proposals:

   **Agent C prompt (breadth-first, per-question):**
   > You are Agent C (Decider) in a PCV Research experiment (breadth-first mode).
   >
   > **Charge file contents:** [insert charge content]
   >
   > **Previously resolved answers:**
   > [Insert all prior resolutions]
   >
   > **Current Question [N]:** [question text]
   >
   > **Agent A proposes:** [A's answer]
   > **Agent B proposes:** [B's answer]
   >
   > Produce:
   > 1. Your resolved answer
   > 2. Brief reasoning
   > 3. Convergence tag: AGREE / DIVERGE / PARTIAL

   c. **Add Agent C's resolution to the shared context accumulator.**

   d. **Save the round artifacts** to `rounds/qN_proposals.md` and `rounds/qN_decision.md`.

5. **Draft the MakePlan** from the accumulated resolutions, following PCV v3.9 MakePlan structure.

6. **Save all artifacts** to the instance directory.

7. **Return**: the MakePlan content, per-question convergence tags, and summary.

**Log each agent invocation to instrumentation.csv.**

> You are a convergence analyst. Two independent depth-first PCV planning runs produced MakePlans for the same charge. Compare them:
>
> **MakePlan from Instance 1:** [content]
> **MakePlan from Instance 2:** [content]
>
> Produce:
> 1. **Convergent elements**: Where both plans agree (high confidence items)
> 2. **Divergent elements**: Where they disagree (uncertainty lives here)
> 3. **Unique contributions**: Items in one but not the other
> 4. **Overall convergence rate**: X of Y substantive decisions agreed
> 5. **Assessment**: Which divergences matter most? Which plan is stronger overall?

Save to `comparison/df_convergence.md`.

### 3b: Breadth-First Convergence

Same analysis for MakePlan-B1 and MakePlan-B2. Save to `comparison/bf_convergence.md`.

Spawn a final comparison agent:

> You are a cross-mode analyst for a PCV Research experiment. Four MakePlans were produced from the same charge file using two different strategies:
>
> **Depth-First** (agents answer all questions independently, then a decider resolves at the end):
> - Instance D1: [summary of convergence tags and key decisions]
> - Instance D2: [summary]
> - Within-mode convergence: [from Step 3a]
>
> **Breadth-First** (agents answer one question at a time, seeing prior resolutions):
> - Instance B1: [summary of convergence tags and key decisions]
> - Instance B2: [summary]
> - Within-mode convergence: [from Step 3b]
>
> Analyze:
> 1. **Mode-level convergence**: Do depth-first plans converge with breadth-first plans? Or do the two modes produce fundamentally different planning approaches?
> 2. **Question-level analysis**: For which questions did the mode matter most? (i.e., depth-first agents answered differently than breadth-first agents)
> 3. **Context dependency**: Did breadth-first agents' later answers noticeably change because they saw earlier resolutions? Evidence of path dependency?
> 4. **Quality assessment**: Which mode produced more internally consistent MakePlans? More comprehensive? More decisive?
> 5. **Efficiency signal**: If one mode consistently converges faster (fewer divergences between instances), that suggests it is more robust.
> 6. **Recommendation**: For a human choosing between these modes, when would you recommend each?

Save to `comparison/cross_mode.md`.

## BEHAVIORAL CONSTRAINTS

- **This command is research-only.** It produces MakePlans for analysis, not for direct use in PCV construction. If the user wants to use a MakePlan, they copy it into their PCV project manually.
- **All agent work runs as subagents** via the Agent tool. The main conversation context stays clean for the human to review results.
- **No file modifications outside the research run directory.** This command never touches the charge file, existing PCV plans, or any project code.
- **Token estimation is approximate.** Character count / 4 is a rough proxy. Note this limitation in all reports. The relative comparisons (DF vs BF, instance 1 vs 2) are valid even if absolute numbers are approximate.
- **Same-model caveat applies.** All agents use the same underlying model. Full convergence does not constitute independent validation. Note this in all comparison reports.
- **Max 8 clarification questions** per PCV v3.9 spec. If the charge needs more, flag it.
- **Question count by deliverable type** (guideline — fewer focused questions improve signal density):
  - Code/architecture: 7-8 (high interdependency)
  - Document/paper: 6-7 (moderate)
  - Presentation/slides: 5-6 (many independent preferences)
  - Analysis/report: 4-5 (scope and framing)
- **CRITICAL — Instances MUST run in the foreground, NOT the background.** Background-spawned agents cannot themselves spawn further subagents via the Agent tool in most Claude Code harnesses. If instances are launched with `run_in_background: true`, each instance will either (a) refuse to do the work and report a blocker, or (b) silently fall back to single-context self-role-play — producing corrupt research data that looks multi-agent but isn't. This was verified empirically on 2026-04-09 and is the primary failure mode of this skill. The orchestration pattern that actually works:
  1. Main conversation generates questions, creates the run directory, and writes `config.md` + `questions_master.md`.
  2. Main conversation spawns **DF Instance 1 in the foreground** (no `run_in_background`). Wait for it to return its MakePlan, convergence tags, and artifact paths.
  3. Main conversation spawns **DF Instance 2 in the foreground**. Wait.
  4. Main conversation spawns **BF Instance 1 in the foreground**. Wait.
  5. Main conversation spawns **BF Instance 2 in the foreground**. Wait.
  6. Main conversation runs Step 3 (within-mode convergence) and Step 4 (cross-mode comparison) — these can use foreground agents as well.
  Wall-clock time is longer than the aspirational "all four in parallel" model, but it is the only way each instance agent retains the Agent-tool access it needs to spawn A/B/C as real subagents.
- **Pre-flight Agent-tool check (MANDATORY).** Before spawning the first instance, the orchestrator MUST confirm the Agent tool is actually available to spawned agents in the current environment. Do this by spawning a **trivial foreground probe agent** with the prompt: "Reply with the single word READY and do nothing else." If this probe returns normally, proceed. If the probe fails or is unavailable, STOP and tell the user: "Agent tool unavailable in this session — PCV-Research cannot run. Restart the session or use `/pcv` directly." Do NOT attempt the protocol without this check.
- **Detect and report single-agent fallback.** If any instance returns without separate A and B perspectives clearly visible in its output (distinct file writes for `agent_a_answers.md` and `agent_b_answers.md`, with substantively different content), it ran in single-agent self-role-play mode. Flag this in the Step 6 report as: "WARNING: Instance [N] ran in single-agent mode. Convergence data is invalid — discard this instance and re-run or reduce to available valid instances." Do NOT silently include corrupt instances in the convergence analysis.
- **Each instance agent is self-contained but foreground.** It receives the charge + questions, internally spawns Agents A, B, and C (using the Agent tool), produces its MakePlan, and returns the complete output. The instance agent needs Agent, Read, Write, Glob, and Grep. It is launched in the foreground so its Agent-tool subagent spawns actually work.
